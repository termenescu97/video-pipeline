# P1 findings

## F-1 [O+C] `retryFile` is non-atomic — crash between writes orphans the file

**Severity**: P1 (Codex) / P2 (Opus) — adopt P1 per Codex's data-safety weighting.
**File**: `lib/services/job_queue_service.dart:306-331`

**Problem.** `retryFile()` makes two separate `_safeWrite` calls — `resetFileForRetry(forceDestDeleteApproved: forceDestDelete)` (line 318) and `requeueJobForFileRetry(file.jobId)` (line 330) — each its own DAO transaction. If the app crashes (power loss, OS kill, isolate exception, abandoned-shutdown drop) between them, the file row is reset to `status=pending && verifyStatus=pending` but the parent `Job.status` stays at `completed`. `getNextQueuedJob` filters out completed jobs; `recoverStaleJobs.getRescuedJobIds` only matches inProgress jobs, jobs with inProgress files, or completed+pending-verify-with-sha256-mode rows — none of these match a pending file in a completed job. The pending row is never picked up.

**Risk.** Operator clicks Retry on a verify-mismatched file in history. Crash mid-write. On next launch the operator sees the job as completed (no warning chip, because the row is now `pending+pending`, neither mismatch nor unverified). The file silently sits in pending state forever. Original bytes on disk remain (possibly corrupt for verifyMismatch). Operator believes the retry is queued. **This is the worst kind of bug for a data-safety release**: silent loss of operator intent, with no observable failure mode.

**Fix sketch.** Wrap both writes in a single Drift transaction inside `retryFile`. Either both writes commit or neither does. The natural shape: a single DAO method (e.g., `JobDao.applyPerFileRetry({required int jobId, required int fileId, required bool forceDestDelete})`) that does the file reset, the job requeue, and the counter recompute in one transaction. Mirror the pattern of `requeueJobForFileRetry` itself, which already wraps status + recompute.

**Verification.** Reproduce by inserting a `throw Exception('synthetic')` between lines 321 and 330 in a test build, click Retry, restart the app. Confirm: (a) before fix, the file is stranded; (b) after fix, the partial mid-Retry state never persists.

# P2 findings

## F-2 [C] Accept mismatch / Accept unverified / Skip mismatch lack the typed-confirmation gate the project's own convention requires

**Severity**: P2.
**Files**: `lib/ui/widgets/job_card_done.dart:321` (Accept mismatched, Accept unverified), `lib/ui/widgets/job_card_active.dart:624` (Skip on the active-card mismatch banner).

**Problem.** These actions use a plain `AlertDialog` with a clickable `FilledButton` for the confirm. The branch's own convention (CLAUDE.md "Load-Bearing Conventions" + the `ConfirmationDialog.showDestructive` / `showCritical` primitives at `lib/ui/widgets/confirmation_dialog.dart`) requires a typed-confirmation gate for trust-lowering decisions. Accept mismatch flips a row that has *cryptographically proven mismatched bytes* to verifyStatus=verified. Accept unverified discards a SHA-256 subsystem failure. Both are at least "destructive" by the project's own taxonomy, arguably "critical" since they immediately unblock chained compression via `maybeChainCompression()`.

**Risk.** A misclick blesses corrupt or unverified bytes and lets HandBrake encode them into the compressed output. Once compressed, the source SD card will get erased (Constitution Principle gives the operator that capability via `eraseEligibilityReason`). The operator's only defense against a misclick is the click-once button — exactly what the typed-confirmation pattern was introduced to prevent.

**Note on prior reviews.** Opus walked past this — it noted "Accept handlers all gate behind AlertDialog with explicit confirm action; no implicit destructive paths" — accepting that *some* dialog exists. Codex caught the *gap between convention and code*: the dialog exists but doesn't match the convention the branch established for trust-lowering operations. This is the kind of gap that diff-focused review catches that broader walkthroughs miss.

**Fix sketch.** Route the three call sites through `ConfirmationDialog.showDestructive` (or `showCritical` if we deem the action catastrophic) with explicit typed phrases — `accept mismatch`, `accept unverified`, `skip mismatch`. The `confirmation_dialog.dart` primitive is already there; it's a UI swap, not a new abstraction.

## F-3 [O] FK constraint `ON DELETE SET NULL` never fires — `PRAGMA foreign_keys = ON` is missing

**Severity**: P2.
**Files**: `lib/database/database.dart` (no PRAGMA set), `lib/database/tables.dart:80` (`Job.parentJobId.references(Jobs, #id, onDelete: KeyAction.setNull)`).

**Problem.** Codex round-2 P2 #4 added `onDelete: KeyAction.setNull` thinking it was load-bearing. SQLite enforces FK actions only when `PRAGMA foreign_keys = ON` is set; Drift's NativeDatabase does NOT enable this by default (verified in drift 2.33.0 sources — drift only QUERIES the pragma, never sets it). There is no `customStatement('PRAGMA foreign_keys = ON')` anywhere in the codebase. **The annotation is dead code.**

**Risk.** Operator deletes a parent transferAndCompress job from history → the chained-compression child still holds `parentJobId = <dead row id>`. `notifyCompressionCompleted` calls `getJob(parentJobId)` which returns null; the code DOES degrade gracefully (parent verify line is omitted), so this is not data loss today. But the documented invariant ("graceful fallback if parent deleted via FK action") is a lie — and any future code that trusts the FK to keep `parentJobId` either-valid-or-null will silently break. Also, `deleteJob` doesn't proactively null the references, so dangling FKs accumulate.

**Why prior rounds missed it.** Codex round-2 added the annotation and 18 subsequent rounds never grepped for the pragma — both Codex and the round-2 reviewer assumed Drift's defaults match SQLite-the-CLI's defaults. They don't.

**Fix sketch.** In `_openConnection` (or in the database's `onCreate`/`onUpgrade` hook), add `await customStatement('PRAGMA foreign_keys = ON');`. With FKs on, validate the v7→v8 migration still applies cleanly (some statements may now fail constraint checks under enforcement). Add a unit test that asserts the pragma is enabled on a freshly-opened database.

## F-4 [O] `maybeChainCompression` has a TOCTOU window between `hasChainedChild` and `_createChainedCompressionJob`

**Severity**: P2.
**File**: `lib/services/job_queue_service.dart:246-290`.

**Problem.** The dedup guard reads `hasChainedChild(parentJobId)` then awaits `getFilesForJob`, then awaits `_createChainedCompressionJob`. Nothing between the check and the create holds a lock or marker. Two concurrent invocations both see `hasChainedChild=false` and both insert. **Reachable today**: operator opens JobCardDone menu, clicks "Accept mismatched", menu closes. Operator opens menu again, clicks "Accept unverified" before the first handler's `maybeChainCompression` completes. Both handlers reach `_createChainedCompressionJob` and both succeed — TWO chained compression children for the same parent.

**Risk.** Doubled compression work, potentially overlapping HandBrake outputs at the same destination path (one would fail or overwrite the other). Wasted cycles, confusing UI (two jobs show up). Not data corruption per se but operator-visible chaos.

**Fix sketch.** Wrap the `hasChainedChild` check + insert in a single transaction (Drift transactions serialize), OR disable the JobCardDone Accept menu items globally while ANY Accept handler is in flight (UI gating similar to "queue running"), OR put a UNIQUE index on `Jobs(parentJobId)` so the second insert fails fast.

## F-5 [O] `startProcessing` race during in-flight `stopProcessing` — two concurrent processing loops

**Severity**: P2.
**File**: `lib/services/job_queue_service.dart:154-208, 215-233`.

**Problem.** `stopProcessing()` sets `_isProcessing=false` immediately and returns the completer future, but the original processing loop only sees `_isProcessing=false` at its next while/if check — which could be hundreds of milliseconds away if a transfer was mid-await. During that gap, `maybeChainCompression` (called from JobCardDone Accept handlers) can fire `startProcessing()`. The reentry guard `if (_isProcessing) return` reads `false` and the second loop spins up. When the original loop's await resolves, it continues its iteration and eventually re-checks `while (_isProcessing)` — which is now `true` (set by the second startProcessing). **Both loops run concurrently against `getNextQueuedJob`.**

**Risk.** Two loops can grab the same paused/queued job, both call `markJobStarted` (idempotent in DB but races on `_currentJobId`), both run `_processTransfer` against the same JobFile rows. Two concurrent robocopies on the same destination. SHA-256 hashes computed twice. `markFileCompleted` race-writes. Realistic trigger: window-close `_gracefulShutdown` Phase B is awaiting the 10s drain timeout, operator (still seeing the UI) clicks Accept on a history card → `maybeChainCompression` → `startProcessing` → two loops while the DB is still open.

**Fix sketch.** Track loop generation via a separate `_stopRequested` flag distinct from `_isProcessing`. The loop only EXITS when it observes `_stopRequested`. New `startProcessing` calls during pending stop should await the existing completer (or be rejected outright until the prior stop drains). Alternatively, gate `startProcessing` on `_stopCompleter == null` rather than `!_isProcessing`.

# P3 findings

## F-6 [O+C] `notifyTransferCompleted` reports "Verified: 0" for clean size-mode transfers

**Severity**: P3 (both reviewers).
**File**: `lib/services/slack_service.dart:67-94`.

**Problem.** Codex round-20 P2 #2 added `parentNotVerifiedFiles` to `notifyCompressionCompleted` so chained Slack pings read "$N size-verified · Passed" for size-mode parents. The same fix was NOT applied to `notifyTransferCompleted`, which still emits `Verified: 0 · Unverified: 0 · Mismatch: 0\nVerification: Size — Passed` for a clean size-mode transfer. The "Verified: 0" line directly contradicts the verdict a line below.

**Risk.** Non-technical operator (target user per CLAUDE.md) cannot reconcile "0 verified" with "Passed" in the same message. Inconsistency between transfer-side and compression-side messaging on the axis Codex round-20 just fixed for compression.

**Fix sketch.** Plumb the size-mode notVerified count to `notifyTransferCompleted` (mirror the compression-side parameter). Render "$N size-verified" instead of "Verified: 0" when verification mode is size and notVerifiedCount > 0. Compute the count in `_processTransfer` and pass it through alongside the existing 3 verify counts.

## F-7 [O] Phase 7 of v8 migration doesn't clear stale `Job.errorMessage` when lifting status from `failed → completed`

**Severity**: P3.
**File**: `lib/database/database.dart:225-238`.

**Problem.** Phase 7 of the v8 migration lifts `Job.status` from `failed` to `completed` for jobs whose only failed children were hash-only (mismatch/unverified) rows. `Job.errorMessage` (set by the original `markJobFailed` to something like "3/5 files transferred, 2 failed copy") is NOT cleared. Lifted jobs now render as `JobStatus.completed` in JobCardDone but still carry a populated `errorMessage` from the failed era.

**Risk.** Any UI surface that reads `job.errorMessage` (Audit/Errors detail tabs, history widgets) shows a stale failure message on a job marked completed. The operator sees "Completed" with "2 failed copy" error text — contradictory and confusing. JobCardDone's dot color logic uses status + verify-axis correctly, but the underlying message remains misleading.

**Fix sketch.** In Phase 7, also `SET error_message = NULL` for the lifted rows (or rewrite to "Migrated from v7: hash failures reclassified as verify warnings"). Alternatively, suppress errorMessage rendering when the verify-axis tally is the only failure mode.

## F-8 [O] Recovery branch's `incrementUnverified` can permanently undercount on shutdown abandon

**Severity**: P3.
**File**: `lib/services/job_queue_service.dart:459-460, 797-798`.

**Problem.** `markFileUnverified` and `incrementUnverified(jobId)` are two separate `_safeWrite` calls. If `_shutdownAbandoned` flips between them (Phase B times out at exactly the wrong moment), the row is `unverified` but the counter isn't bumped. Recovery does NOT re-rescue this job (rescued set requires `verifyStatus=pending` for the third arm — `verifyStatus=unverified` is never rescued). `recomputeCountersFromFiles` runs only for rescued jobs. The counter under-counts indefinitely until the next operator action triggers `_recomputeUnverifiedForFile` or `requeueJobForFileRetry`.

**Risk.** `Job.unverifiedFiles` silently lags reality. UI badges show "0 unverified" while a file row carries the unverified chip. Operator might not notice the warning. Self-heals on next operator action, but the gap is observable.

**Fix sketch.** Combine `markFileUnverified` + `incrementUnverified` into a single transactional DAO method (mirror the pattern of `acceptUnverified` which already does this). Apply to both forward path (line 797-798) and recovery branch (line 459-460).

## F-9 [O] Transfer staging dirs (`.tmp_robocopy_<tag>`) leak forever on crash between rename and dir cleanup

**Severity**: P3.
**File**: `lib/services/transfer_service.dart:121-160`.

**Problem.** For renamed-destination transfers, the staging dir is created at `<destDir>/.tmp_robocopy_<microsecond-tag>` and removed after `staged.rename(destinationFile)` succeeds. There is no startup janitor that scans for orphaned `.tmp_robocopy_*` dirs. If the app crashes between the rename (line 133) and the cleanup (line 154), the empty staging dir stays on disk forever. Subsequent transfers create new tagged dirs.

**Risk.** Long-term filesystem litter on the destination drive. Operator sees `.tmp_robocopy_*` dirs proliferating in their video archive. Cosmetic but noticeable; could trigger antivirus scans on every directory listing, slowing the operator's file-explorer access. Not data loss.

**Fix sketch.** On startup (next to `recoverStaleJobs`), walk known destination roots from `Job.destinationPath` and remove any empty `.tmp_robocopy_*` subdirs. Or stamp the tag with the PID + a marker file so a startup janitor can prove the dir is not from a currently-running instance.

## F-10 [O] `_processTransfer` SHA-256 mode credits bytes IMMEDIATELY but size mode credits AFTER `verifyTransfer` — inconsistent ordering invites future regression

**Severity**: P3.
**File**: `lib/services/job_queue_service.dart:756-763` (SHA-256 path) vs `:849-882` (size path).

**Problem.** SHA-256 path: `markFileCompleted(verified=false)` → `updateJobProgress` → THEN verify. Comment explicitly says this is structural protection against the v2.4.0 0/27 freeze (CLAUDE.md "v8 (017A) Load-Bearing #2"). Size-mode path: `verifyTransfer` (size check) → `markFileSizeOnlyVerified` → `updateJobProgress`. Two different sequences for "credit bytes vs verify" between modes. The convention claims the SHA-256 ordering is load-bearing, but size-mode silently violates it.

**Risk.** A future refactor that uses size-mode as the template for "how should this work" will introduce a regression equivalent to the 0/27 freeze. Also: if `verifyTransfer` ever blocks (e.g., a slow filesystem stat over SMB), bytes are never credited until it returns, freezing progress on size-mode jobs in a way SHA-256 mode cannot.

**Fix sketch.** Apply the "credit bytes immediately after robocopy success" pattern to size-mode too — call `markFileSizeOnlyVerified` upfront, do the size check, write a status only on FAILURE. Or document the ordering inconsistency as deliberate (which it doesn't appear to be) and add a load-bearing comment.

# Walked-through-but-found-no-bugs (audit trail)

Both reviewers covered (and signed off on) these areas. Recording for traceability so the next adversarial pass doesn't re-walk them blind:

- **PowerShell argv shape** — `process_runner` length-3 assert; `ps_escape` doubles single quotes; smart-quote false-positive documented.
- **v7→v8 migration phase ordering** — Phase 4b's `status='failed' → 'completed'` for mismatch/unverified rows is downstream of Phase 3+4's verify_status assignment; Phase 5's copy-error backfill correctly skips already-classified rows; Phase 6 recompute scoped to affected jobs.
- **`forceDestDeleteApproved` lifecycle** — consumed at top of pending-row iteration before any branch; survives crash by design.
- **`JobFile.startedAt` preservation across resets** — confirmed in `resetFileToPending`, `resetFileForRetry`, `recoverStaleJobs`.
- **Phased shutdown structure** — Phase C cleanup runs unconditionally regardless of Phase B outcome.
- **Case-collision normalization** — both batch and single-job paths run `normalizeCaseCollisions`; rebuilt-after-newFolder paths re-run it; `_suffixedPathAgainst` checks both disk and lowercased planned-set.
- **Per-file retry blast radius** — `status=failed` early-skip in `_processTransfer` correctly isolates per-file retry from job-level retry.
- **Auto-chain compression gating** — `maybeChainCompression` checks all gates (status in {completed, paused}, no non-completed file rows, no mismatch/unverified rows). The TOCTOU finding above is a separate concern from the gate-evaluation correctness.
- **Constitution Principle I (human-in-the-loop)** — confirm dialogs exist on all destructive actions. The Accept-typed-gate finding (F-2) is a *convention-conformance* gap, not a *Principle I* violation.

# Suggested spec-kit framing

When this becomes a spec-kit feature (likely "018-pre-tag-hardening"), the natural shape:

- **One requirement per finding.** Don't bundle F-3 (FK pragma) with F-4 (chain TOCTOU); they're independently testable.
- **User stories.** US1 = "Operator clicks Retry on a mismatched file and the retry survives a crash". US2 = "Operator clicks Accept and is forced to type to confirm". US3 = "Operator's Slack messages match what actually happened". US4 = "Two simultaneous Accept clicks produce one chained job, not two". US5 = "Closing the window during processing doesn't spawn a second processing loop".
- **Success criteria.** F-1 → unit test that synthetically interrupts mid-`retryFile` and verifies the file isn't stranded. F-3 → unit test that asserts `PRAGMA foreign_keys = ON` after `_openConnection`. F-4 → integration test that fires two `maybeChainCompression` calls back-to-back and asserts only one child exists. F-5 → integration test that overlaps `stopProcessing` and `startProcessing` and asserts only one loop ever runs.
- **Bundle rationale.** All P1 + P2 findings ship together (per the operator's "bundle deferred fixes before asking for QA" preference). P3 findings can also ride along — they're cheap. Total scope estimate: ~10 source-file edits, 4-6 new unit tests, 1 integration test for the chain-TOCTOU case.
- **Codex review cadence.** Per `feedback_adversarial_review.md`, plan + tasks + post-implement reviews. The post-implement Codex pass is especially important here because we're fixing race conditions, which prior rounds missed precisely because they're hard to spot in static review.
