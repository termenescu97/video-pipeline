# Implementation Plan: UI/UX Redesign — Visual Hierarchy & Operator Trust

**Branch**: `014-ui-redesign` | **Date**: 2026-05-07 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/014-ui-redesign/spec.md`

## Summary

Comprehensive UI/UX redesign of v2.3.0 — same code engine, new shell. Three-column layout (Sources / Queue+Detail / Activity) replaces master-detail. Status bar replaces bare AppBar. Job cards split into four variants (Active / Next-up / Queued / Done). Detail expands inline with three tabs (Files / Audit / Errors). Create Job collapses to common path with live plan summary. Theme foundation centralizes spacing, typography, status colors. No data model changes; no schema migration; no new external dependencies.

## Technical Context

**Language/Version**: Dart 3.x / Flutter 3.x
**Primary Dependencies**: window_manager (already), tray_manager (already), file_picker (already), drift (already), package_info_plus (already from 013). One new asset: JetBrains Mono font.
**Storage**: SQLite via Drift, schema v5 (unchanged)
**Testing**: Manual QA on Windows 11 (flutter analyze + manual test plan from quickstart.md)
**Target Platform**: Windows 11
**Project Type**: Desktop app
**Performance Goals**: Files lists with 200+ entries scroll at 60fps; status bar updates within 100ms of state change; layout does not shift when live numbers update (tabular figures)
**Constraints**: Single Windows .exe; min window 1280×720 (up from 800×600); Material 3 with seeded blue (no rebrand); no dark mode in this feature; no new external runtime
**Scale/Scope**: ~12 UI files modified, ~10 new widgets created, ~1 font asset added, ~600 lines of theme/layout, ~1500 lines of widget rewrites and replacements

## Implementation Phases

### Phase A: Theme foundation (no behavioral changes)

These are isolated edits to the theme layer; nothing visual changes yet, but every screen will re-read these values once we touch them in later phases.

**A1. Spacing scale** (FR-037)
- New file: `lib/ui/theme/insets.dart`
- Define `class Insets { static const xs = 4.0; s = 8.0; m = 12.0; l = 16.0; xl = 24.0; xxl = 32.0; }`
- Codify the values pulled from the existing literals (8 / 16 / 24 / 32 dominate today).

**A2. Text scale with tabular figures** (FR-038)
- New file: `lib/ui/theme/text_styles.dart`
- Define `AppTextStyles.display / headline / title / body / caption` returning `TextStyle` with appropriate sizes and `fontFeatures: [FontFeature.tabularFigures()]` on display and any numeric-heavy variant.
- Wire into `app_theme.dart` `textTheme:` overrides for the matching Material 3 slots (`displayMedium`, `titleLarge`, `bodyLarge`, etc.) so `Theme.of(context).textTheme` returns these.

**A3. StatusColors adoption** (FR-039)
- The extension exists in `app_theme.dart:50-56`. It's defined but barely used.
- Audit every raw `Colors.red` / `Colors.orange` / `Colors.green` in screens & widgets (29 occurrences across 9 files per the inventory). Replace with `Theme.of(context).extension<StatusColors>()!.error / warning / success`.
- Add `StatusColors.dot.idle / active / recentDone / attention / warning` getters used by the status bar.

**A4. Density compact + JetBrains Mono asset** (FR-040, FR-041)
- Bundle `assets/fonts/JetBrainsMono-Regular.ttf`. Declare in `pubspec.yaml` `flutter > fonts:`.
- Add `fontFamily: 'JetBrainsMono'` to a new `AppTextStyles.mono` style.
- Set `visualDensity: VisualDensity.compact` in `ThemeData` (both light and dark, even though dark is unused this release).

**A5. Min window size** (FR-002)
- `lib/main.dart`: change `setMinimumSize(const Size(800, 600))` to `setMinimumSize(const Size(1280, 720))`.
- Update `_AlreadyRunningApp` second-instance window to keep its compact 520×320.

### Phase B: Shell + status bar (structural)

**B1. StatusBar widget** (FR-003, FR-004)
- New file: `lib/ui/widgets/status_bar.dart`
- Renders: app icon, app name, state dot (animated pulse on blue/orange/red), queue summary text with time-of-day completion when available, operator name (read live from `settingsDao.watchSettings()`), settings IconButton, `?` IconButton.
- Subscribes to `jobQueueService.progressNotifier` + `jobDao.watchAllJobs()` to compute the state dot color and summary.
- **State dot precedence** (when multiple conditions are true, worst wins): `red` (any failed job in queue) > `orange` (Slack misconfigured OR HandBrake missing) > `blue` (a job is running) > `green` (queue just cleared, within 5min) > `grey` (idle). Summary text describes the highest-priority condition.
- Tray tooltip mirror: pushes the same summary text to `trayManager.setToolTip()` whenever it changes (throttled to 1Hz).
- "Recent done" timer: a single `Timer(Duration(minutes: 5))` started when the queue empties; cancelled on next user action (job created, queue started). Drives the green→grey transition.

**B2. ShellScreen rewrite** (FR-001)
- Three columns: `Row(children: [SourcesPanel(width: 240), Expanded(child: HomeScreen()), ActivityPanel(width: 300)])`.
- Status bar replaces AppBar via `Scaffold(appBar: PreferredSize(preferredSize: Size.fromHeight(56), child: StatusBar()))`.
- All existing keyboard `Shortcuts` + `Actions` extended to the new shortcut list (FR-052) — each new shortcut maps to an existing Intent or new one.
- WindowListener implementation from 013 stays (graceful shutdown).
- The right pane stops being a navigation target; `CreateJobScreen` shows as an inline replacement of the central column when active; `JobDetailScreen` no longer routed by default (detail is inline in `JobCardActive`). The route registration is kept for backwards compatibility.

### Phase C: Sources panel (left column)

**C1. SourcesPanel widget** (FR-020, FR-021, FR-022, FR-023)
- New file: `lib/ui/widgets/sources_panel.dart`
- Polls `driveService.getRemovableDrives()` every 3 seconds via `Timer.periodic` started in `initState`. Cancelled in `dispose()`. Transient errors (drive unplugged mid-poll) are swallowed silently — the next tick recovers.
- Shows: each detected drive as a tappable row (drive letter, label, capacity used/total, free-space pill).
- Tapping a row calls `widget.onSourceSelected(drive)` which the shell turns into showing Create Job pre-filled.
- Empty state: pulsing "Listening for SD cards…" using a slow `AnimationController` (also disposed in `dispose()`).
- Loading state: 3 skeleton rows (only on the very first poll before any result has arrived).

**C2. Wire SourcesPanel into ShellScreen**
- Pass an `onSourceSelected` callback that updates shell state to show Create Job pre-filled.
- Replace the inline `DriveList` block in `CreateJobScreen` with a simple "Selected source: …" display (DriveList becomes used only for compression-only "Folder…" picker fallback, or removed entirely if not needed).

### Phase D: Activity panel (right column)

**D1. ActivityPanel widget** (FR-031, FR-032)
- New file: `lib/ui/widgets/activity_panel.dart`
- Subscribes to `jobDao.watchCompletedJobs()`.
- Groups by day: today / yesterday / "this week" (≤7 days) / older. Day boundary uses local time.
- Each entry shows compact `JobCardDone`.
- Bottom of panel: prominent `FilledButton.tonalIcon` with download icon labeled "Export CSV" calling existing CSV export logic.
- Empty state: brief explanation message ("Completed jobs will appear here.").

### Phase E: Job card variants + queue treatment

**E1. Variant widgets** (FR-005, FR-005a, FR-006, FR-007, FR-008, FR-009)
- New files: `job_card_active.dart`, `job_card_next_up.dart`, `job_card_queued.dart`, `job_card_done.dart`.
- `job_card.dart` becomes a router: builds the right variant based on `job.status` and queue context.
- All four variants share: state dot at left edge (12px), left-border color from `StatusColors`, type glyph (monochrome icon), `⋯` overflow with all context-menu actions.
- Active variant: large card; embedded `PipelineProgressBar`; current filename on its own line (middle-ellipsis); stats line including **verification badge** (`SHA-256 ✓` / `Size-only`); phase indicator strip; Pause / Cancel / `⋯` action buttons; expandable into `DetailTabs` inline.
- Next-up variant: same height as Active, no progress bar; "Press Start to begin" hint replacing stats line; `Start` and `Cancel` action buttons.
- Queued variant: 64px slim row; drag handle `☰` on right; click body to expand inline.
- Done variant: dimmed 48px row, no actions other than `⋯`.

**E2. Drag handle wiring** (FR-006)
- Move `ReorderableDragStartListener` from the whole `JobCard` to a small wrapper around the `☰` icon in `JobCardQueued`.
- `JobCardActive` is positionally fixed and not reorderable.
- `JobCardNextUp` IS reorderable — it's just the styled first-queued job. Dragging another card above it promotes that card to next-up; dragging Next-up below another card demotes it. The "next-up" styling is purely positional (always renders for `queue.first`).

**E3. Failure grouping** (FR-011)
- A "N failed — review" banner pins to the top of the queue panel when failed jobs exist, with [Retry all] [Dismiss] actions.
- Failed jobs themselves remain in their normal queue position with red dots — they are NOT re-sorted to the top. The Active/Next-up hero card stays the visual anchor (FR-005a).
- The banner is the surfacing mechanism; sort order is preserved so the operator's mental model of queue position holds.

**E4. Completion celebration** (FR-012)
- When the queue transitions from "running" to "all done" state, show a `JobCardCompleted` ephemeral card in the active slot for the duration of the green dot (5min or until next operator action). Two CTAs: `Erase Cards` and `New Job`.
- **Erase Cards CTA semantics** (Constitution Principle I): launches a sequential per-card flow — for each detected card from the just-completed batch, the existing typed-confirmation dialog (`_showEraseConfirmDialog`, serial-number identity check, typed drive-path field) fires once. The CTA never bulk-erases multiple drives behind a single confirmation. Operator can cancel mid-sequence; already-erased cards stay erased.

### Phase F: Detail tabs (inline)

**F1. DetailTabs container** (FR-013, FR-014)
- New file: `lib/ui/widgets/detail_tabs.dart`.
- TabBar with three tabs: Files (count) / Audit / Errors (count).
- Tabs always visible. Errors tab label shows `Errors (N)` where N is failed-file count; can be 0.

**F2. FilesTab** (FR-015, FR-016, FR-017)
- New file: `lib/ui/widgets/files_tab.dart`.
- Single `ListView.builder` for the whole tab — no nested scrollers. Item index 0 is the filter chip row (sticky via `SliverPersistentHeader` if needed, otherwise pinned at top of the list); subsequent indices are file rows. This avoids the `shrinkWrap` + `SingleChildScrollView` nesting trap entirely.
- Filter chip row: All / Pending / In progress / Completed / Failed. Selected filter is component-local state; changing filters triggers a rebuild that re-filters the file list backing the builder.
- Per-row: status icon, filename (middle-ellipsis), size, "✓ matches" badge for verified files.
- Hash popover: `showMenu` or `showDialog` opening a small panel with both hashes in `JetBrainsMono` and a "Copy both" button using `Clipboard.setData`.

**F3. AuditTab** (FR-014)
- New file: `lib/ui/widgets/audit_tab.dart`.
- Renders job timeline: created → started → file events → completed; verification mode; operator name; total bytes; full hash list.

**F4. ErrorsTab** (FR-014)
- New file: `lib/ui/widgets/errors_tab.dart`.
- Lists failed files with their errorMessage. Empty state copy: "No errors. Every file completed successfully."

**F5. Erase as header action** (FR-018, FR-019)
- Move the existing `_eraseSourceDrive()` flow from inside `job_detail_screen.dart` to a header action button in the active card.
- Button is always visible for transfer-type jobs whose source is a removable drive; disabled with reason text ("Waiting for verification" / "Job not yet complete") until eligible.
- Existing safety gates (`_showEraseConfirmDialog`, serial-number identity check, typed drive-path field, size-only warning) remain unchanged in behavior — they get refactored out of `job_detail_screen.dart` into a new `lib/ui/widgets/erase_drive_action.dart` for reuse.

### Phase I: Settings side-nav

Test result line "Last test: OK 11:42" / "Last test: failed at 11:42" lives in widget state only (in-memory, resets on app restart — see data-model.md). On launch the line reads "Untested". ### Phase J: Standardized destructive dialogs + conflict tweaks

**J1. ConfirmationDialog severity-aware variant** (FR-047)
- Update `confirmation_dialog.dart` to support a `severity: ConfirmationSeverity` (info / destructive / critical).
- Critical severity adds a typed-confirmation field (configurable text — defaults to action verb).
- Existing call sites updated. Erase SD card uses the existing `_showEraseConfirmDialog` (already typed-confirmation); destructive ops like "Clear history" or "Delete job" route to ConfirmationDialog with `destructive` severity (no typed gate, but red emphasis).

**J2. ConflictResolutionDialog tweaks** (FR-046)

### Phase K: Keyboard shortcuts + cheat sheet

**K1. Expand shortcuts in ShellScreen** (FR-052)
- Map all shortcuts in the `Shortcuts(shortcuts: {...})` block.
- Wire each via `Actions(actions: {...})` to existing methods or new ones:
  - `Ctrl+N` → existing CreateJobIntent
  - `Ctrl+Shift+C` → opens `CopyAllCardsDialog`
  - `Ctrl+Enter` → existing PauseResumeIntent
  - `Ctrl+,` → navigates to settings
  - `Ctrl+E` → triggers existing CSV export from Activity panel
  - `Ctrl+L` → opens log file in Explorer with the file selected: `Process.run('explorer', ['/select,', logService.logPath])`
  - `↑/↓` → moves `selectedQueueJobId` up/down in the queue list
  - `Space` → toggles inclusion in `expandedJobIds` for the selected card
  - `Delete` → deletes selected job (after standard ConfirmationDialog)
  - `Ctrl+R` → retries selected job if status is failed
  - `?` / `F1` → opens KeyboardCheatSheet modal
- Make sure `Shortcuts` is wrapped above any `Focus` / `TextField` so typing in inputs doesn't trigger them. (Flutter's `Shortcuts` widget has built-in scoping — the inputs already capture key events.)

**K2. KeyboardCheatSheet modal** (FR-053)
- New file: `lib/ui/widgets/keyboard_cheat_sheet.dart`.
- Modal listing all shortcuts grouped by category (Job Management / Queue Control / Navigation / Help).
- Triggered by `?` (Question intent), `F1`, or the `?` icon in StatusBar.
- Dismiss on `Esc` or any outside click.

### Phase L: Empty / loading / error states

**L1. SkeletonRow widget** (FR-049)
- New file: `lib/ui/widgets/skeleton_row.dart`.
- Single shimmering placeholder row at configurable height.
- Used by `SourcesPanel` (3 rows during initial drive scan), home queue (3 rows during initial job query), `FilesTab` (5 rows).

**L2. Hero empty state with cards detected** (FR-048)
- When queue is empty AND `driveService.getRemovableDrives()` returns ≥1 drive, the queue panel shows a hero card: "N cards detected — [Copy All Cards →]" with the Copy All Cards button as the primary affordance.

**L3. HandBrake-missing banner on home** (FR-050)
- The HandBrake-not-installed banner currently in `create_job_screen.dart:109-133` becomes a shared widget `lib/ui/widgets/handbrake_banner.dart` and renders on the home queue panel as well, alongside the existing Slack-unconfigured banner.

**L4. "Recovered after restart" indicator** (FR-051)
- Add a small chip on `JobCardQueued` and `JobCardNextUp` for jobs whose recovery happened on the most recent app launch — drives this off a per-process flag set in `JobDao.recoverStaleJobs()` (the dao caches the IDs of recovered jobs in memory; the card reads from this list).
- **Clearing semantics**: an ID is removed from `recoveredJobIds` only when the operator acts on *that specific recovered job* (resume / cancel / delete / retry). Creating an unrelated new job does NOT clear other jobs' recovery chips — operators may queue work without first reviewing every recovered item, and we don't want the warning to vanish silently.

## Risk Assessment

| Change | Risk | Mitigation |
|--------|------|------------|
| Three-column layout at 1280×720 minimum | Medium — may feel cramped on the team's actual monitor; sources column might get too narrow when SD card labels are long | Manual QA on the team's actual Windows machine before merge; if 240px is too narrow, bump to 280 |
| StatusColors adoption across 9 files | Low — mechanical refactor; analyzer catches any miss | Run `flutter analyze` after each file batch |
| Status bar tray tooltip mirroring | Medium — push frequency could throttle tray manager | Throttle updates to 1Hz |
| Inline detail expansion vs route | Medium — existing JobDetailScreen route may have callers we miss | Keep the route registered; deprecate but don't delete in this feature; smoke-test all paths |
| Drag handle change moves listener from card to icon | Low — well-bounded refactor | Test drag in queue with mixed variants |
| Plan summary live computation in Create Job | Low — existing scan logic just feeds a panel instead of dialog | Reuse scan output |
| Keyboard shortcut focus scoping | Medium — must not fire while typing in text fields | Verify by typing `?` in operator-name field; should insert "?" not open cheat sheet |
| JetBrains Mono asset bundling | Low — well-supported in Flutter | Confirm font renders on Windows build |
| Min window bump to 1280×720 | Medium — operators may have used a smaller window on the prior version | Document in changelog; window_manager will snap to new minimum |
| 5-minute green-dot timer | Low — single Timer per queue cycle | Cancel on dispose; reset on new job creation |
