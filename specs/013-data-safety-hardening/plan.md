## Summary

Fix 14 validated findings from the GPT 5.5 adversarial code review covering data safety (silent overwrites, cross-card collisions), crash recovery (stranded jobs, partial creation), subprocess reliability (pipe hangs, uncancellable hashing, unsafe shutdown), instance safety, queue ordering, and operational correctness (PowerShell error handling, version metadata).

No new UI screens. No database schema migration. All changes are behavioral fixes to existing code paths.

## Implementation Phases

### Phase A: Foundation Fixes (no UI changes, lowest risk)

These are isolated fixes that don't affect each other and can be implemented in any order.

**A1. Fix ProcessRunner stream draining** (FR-009)
- File: `lib/utils/process_runner.dart`
- Change: Replace `Future<void>.value()` with `process.stdout.drain()` / `process.stderr.drain()` when no callback provided
- Lines: 17-31
- Risk: Low — drain() is a no-op if stream is empty

**A2. Fix instance lock** (FR-012)
- File: `lib/utils/instance_lock.dart`
- Changes:
  - Import `dart:io` and use `pid` (the global property) instead of undefined `$pid`
  - Change `catch (_) { return true; }` to `catch (_) { return false; }` (fail closed)
  - Move lock file path to `getApplicationSupportDirectory()` instead of executable directory
  - **Atomic write pattern**: write PID to a temp file (`copiatorul3000.lock.tmp`), then rename to `copiatorul3000.lock`. Rename is atomic on both Windows and POSIX, preventing two processes from both passing the exists-check simultaneously.
  - **Stale lock recovery**: if lock file exists and PID is not running (checked via `tasklist`), delete stale lock and re-acquire. If PID check itself fails, treat lock as held (fail closed).
- Lines: 17-35

**A3. Fix PowerShell error handling** (FR-015, FR-016)
- File: `lib/services/drive_service.dart`
- Changes:
  - Add `_runPowerShell(List<String> args)` helper with try/catch returning `ProcessResult?`
  - Refactor `_getWindowsDrives()`, `getDiskFreeSpace()`, `getDriveIdentity()` to use helper
  - In `getDriveIdentity()`, use `$args[0]` pattern instead of string interpolation
  - **User-friendly error propagation**: when `_runPowerShell()` returns null (PowerShell unavailable or crashed), each caller returns a typed failure:
    - `_getWindowsDrives()` → returns empty list (no drives shown, existing "no drives detected" UI message covers this)
    - `getDiskFreeSpace()` → returns -1 (existing "unknown" display covers this)
    - `getDriveIdentity()` → returns null (callers already handle null identity with fallback text)
  - Add `getDriveIdentity()` serial number field: query `Win32_DiskDrive` via `Win32_LogicalDiskToPartition` + `Win32_DiskDriveToDiskPartition` associations to get disk serial number. Return `({String label, int totalBytes, String? serialNumber})`. This provides a stable physical device identifier for erase re-verification (D1).
- Lines: 37-140

**A4. Fix version metadata** (FR-017)
- Files: `lib/utils/constants.dart`, `lib/services/update_service.dart`, `pubspec.yaml`
- Changes:
  - Remove `appVersion` from constants.dart
  - Add `package_info_plus` to pubspec.yaml
  - Update `lib/services/update_service.dart` to read version from `PackageInfo.fromPlatform()`
  - Set pubspec.yaml version to `2.3.0+1`

### Phase B: Database & Queue Fixes

- File: `lib/database/daos/job_dao.dart`
- Add `recoverStaleJobs()`: transaction that updates all inProgress jobs → paused, all inProgress files → pending
- File: `lib/main.dart`
- Call `await jobDao.recoverStaleJobs()` after DB init and after instance lock acquisition, before `runApp()`

**B2. Transactional job creation** (FR-005)
- File: `lib/database/daos/job_dao.dart`
- Add `createJobWithFiles(JobsCompanion, List<JobFilesCompanion>, int totalFiles, int totalBytes)` using `transaction()`
- File: `lib/ui/screens/create_job_screen.dart`
- Replace three separate calls with single `jobDao.createJobWithFiles()`
- File: `lib/services/job_queue_service.dart`
- Replace three separate calls in `createBatchTransferJobs()` with `jobDao.createJobWithFiles()`

**B3. Fix queue ordering** (FR-013, FR-014)
- File: `lib/database/daos/job_dao.dart`
- `getNextQueuedJob()`: change `orderBy` from `createdAt` to `sortOrder ASC, createdAt ASC`
- Add `getMaxSortOrder()`: returns max sortOrder from active jobs (returns 0 if no jobs)
- File: `lib/ui/screens/create_job_screen.dart`
  - At single-job creation: `sortOrder = await jobDao.getMaxSortOrder() + 1`
- File: `lib/services/job_queue_service.dart`
  - In `createBatchTransferJobs()`: compute `baseOrder = await jobDao.getMaxSortOrder()` once before the loop, then assign `baseOrder + 1`, `baseOrder + 2`, etc. per card in the loop. This ensures each batch job gets a unique, sequential sortOrder.

### Phase C: Transfer Safety

**C1. Per-card subfolders in batch copy and single-job drive root** (FR-001)
- **Depends on**: A3 (uses refactored `getDriveIdentity()`)
- File: `lib/services/job_queue_service.dart`
- In `createBatchTransferJobs()`:
  - Call `driveService.getDriveIdentity(drivePath)` for each drive
  - Sanitize label (replace non-alphanumeric with `_`, empty → `Drive`)
  - Construct subfolder: `${sanitizedLabel}_${driveLetter}`
  - Prepend subfolder to relative paths: `p.join(destination, subfolder, relativePath)`
  - Store the full subfolder path in job's `destinationPath` for clarity
- File: `lib/ui/screens/create_job_screen.dart`
  - When creating a single job from a drive root (source is a removable drive), apply the same `label_driveletter` subfolder logic. This satisfies US1 scenario 2 (single job from drive root with prior same-structure destination).

**C2. Destination conflict detection** (FR-002)
- New file: `lib/ui/widgets/conflict_dialog.dart`
  - `ConflictResolutionDialog` widget with options: Skip existing, Rename (auto-suffix `_1`, `_2`), Choose new folder, Overwrite (typed "OVERWRITE"), Cancel
  - Returns enum: `skip`, `rename`, `newFolder`, `overwrite`, `cancel`
- File: `lib/ui/screens/create_job_screen.dart`
  - After building file list, check `File(dest).existsSync()` for each entry
  - If conflicts found, show `ConflictResolutionDialog`
  - Handle each resolution: filter files (skip), rename conflicting destinations (rename), re-pick folder (newFolder), proceed (overwrite), abort (cancel)
  - **Zero-file guard**: after skip/rename filtering, if the resulting file list is empty, show a message "All files already exist at destination — no files to transfer" and do NOT create a job. This prevents phantom zero-file jobs.
- File: `lib/services/job_queue_service.dart`
  - In `createBatchTransferJobs()`: perform **one global conflict preflight** across ALL cards before creating ANY jobs. Build the complete file list for all cards (with per-card subfolders from C1), check all destination paths, then present conflicts to the caller. Only after resolution, create jobs for cards that still have files to transfer. Skip cards whose files were all filtered out.
  - Return `({int created, int skipped, List<String> conflicts})` to caller for UI handling

**C3. Fix chained compression path flattening** (FR-003)
- File: `lib/services/job_queue_service.dart`
- In `_createChainedCompressionJob()` line 452:
  - Replace `p.join(outputPath, f.fileName)` with `p.join(outputPath, relativePath)`
  - Compute `relativePath` as `p.relative(f.destinationFilePath, from: transferJob.destinationPath)`

### Phase D: Erase Safety

**D1. Erase re-verification and typed confirmation** (FR-006, FR-007)
- **Depends on**: A3 (uses `getDriveIdentity()` with serial number)
- File: `lib/ui/screens/job_detail_screen.dart`
- In `_eraseSourceDrive()`:
  - Store pre-dialog identity (label, totalBytes, serialNumber)
  - After confirmation dialog returns, re-call `getDriveIdentity(drivePath)`
  - Compare **serial number first** (strongest physical identifier), then label + totalBytes as fallback if serial is unavailable. If any mismatch, show "Drive changed — erase aborted" and return
  - Add `TextField` to confirmation dialog requiring operator to type drive label or path to proceed

**D2. Size-only verification warning** (FR-008)
- File: `lib/ui/screens/job_detail_screen.dart`
- In the erase confirmation dialog (inside `_eraseSourceDrive()`):
  - Check `job.verificationMode == VerificationMode.size`
  - If true, add a prominent orange warning inside the dialog body: "Files were verified by size only, not content hash. Proceed with caution."
  - The warning is shown inside the confirmation dialog at the destructive decision point, per FR-008 and spec acceptance scenario US5.3

### Phase E: Subprocess & Shutdown

**E1. Cancellable SHA-256 hashing** (FR-010)
- **Depends on**: A1 (ProcessRunner stream fix must be in place first)
- File: `lib/services/transfer_service.dart`
  - Add `_hashRunner = ProcessRunner()` field
  - Rewrite `computeFileHash()` to use `_hashRunner.run()` with stdout callback capturing hash
  - Update `cancel()` to also kill `_hashRunner`
- File: `lib/services/job_queue_service.dart`
  - After hash cancellation, mark file as `pending` (needs re-verification on resume)
  - Ensure the pending-state write completes before returning from the cancellation path (important for E2 shutdown ordering)

**E2. Graceful shutdown for window close and tray quit** (FR-011)
- **Depends on**: E1 (cancellable hashing must be in place so shutdown can kill hash processes)
- File: `lib/services/job_queue_service.dart`
  - Make `stopProcessing()` return `Future<void>` via a `Completer` that resolves when the processing loop exits and all state persistence writes (including pending-state from E1 hash cancellation) have completed
  - The completer resolves AFTER the loop body finishes its current iteration (including any file status writes), not after just setting the flag
- File: `lib/ui/screens/shell_screen.dart`
  - Add `with WindowListener` mixin to `_ShellScreenState`
  - In `initState()`: `windowManager.setPreventClose(true)`, `windowManager.addListener(this)`
  - Implement `onWindowClose()`: call `_gracefulShutdown()` then `windowManager.destroy()`
  - Update `_gracefulShutdown()` — used by BOTH window close AND tray quit:
    - `await jobQueueService.stopProcessing()` — no timeout on the stop itself (state persistence must complete)
    - Wrap the entire shutdown in a 30-second safety timeout (prevents infinite hang if something is truly stuck). If timeout fires, force-close anyway (better than zombie process).
    - Then close log, release lock, close DB (in order)
    - Remove `exit(0)` — let `windowManager.destroy()` handle it for window close; tray quit calls `_gracefulShutdown()` then `exit(0)`
  - Wire tray quit handler to call the same `_gracefulShutdown()` method (currently it already does, but the method itself is being rewritten)

## Risk Assessment

| Change | Risk | Mitigation |
|--------|------|------------|
| Per-card subfolders | Medium — changes destination path structure for batch AND single-drive jobs | Manual test with 2+ cards; verify single-card behavior; verify single-job from drive root creates subfolder |
| Conflict detection | Medium — new UI dialog in critical path; global batch preflight adds complexity | Test: empty dest, partial conflict, full conflict (skip-all → no job created), batch multi-card, single job, rename suffix |
| Startup recovery | Low — runs once at startup, idempotent | Transaction ensures atomicity; runs after lock acquisition; test with forced kill during transfer |
| Transactional creation | Low — wrapping existing calls | Functionally identical to current flow, just atomic; zero-file guard prevents phantom jobs |
| Instance lock fix | Low — isolated utility | Atomic rename pattern; fail closed on errors; test: normal launch, double launch, stale lock, read-only FS |
| Shutdown sequence | Medium — timing-dependent; covers window close + tray quit | State persistence guaranteed before DB close; 30s safety timeout; test both close paths with active transfer + hashing |
| Erase re-verification | Medium — serial number WMI query adds complexity | Fallback to label+size if serial unavailable; test with card swap during dialog; typed confirmation required |
| Queue ordering | Low — single query change + batch increment | Batch computes max once, increments per card; verify drag-reorder + processing order match |
| ProcessRunner drain | Low — additive, no behavior change for existing callers | drain() is no-op on empty streams |
