## CONVERGENT P1 (data loss / Constitution-I violation)

### F-1 ‚Äî Drive-letter remap silently transfers from the WRONG SD card, then erases it

- **Opus**: P1-1 LIKELY ¬∑ **Codex**: P1 #1 LIKELY
- **Files**:
  - `lib/database/tables.dart:42` (Job table ‚Äî no source identity column)
  - `lib/services/job_queue_service.dart:417` (only `Directory.exists()` check on resume)
  - `lib/ui/widgets/erase_drive_action.dart:133` (identity captured per-letter at erase time only)
- **Failure mode**: A paused/queued/recovered transfer job persists `sourcePath = 'E:\\'` and per-file absolute paths embed the drive letter. On app relaunch (or pause-then-resume), if Card A has been ejected and a DIFFERENT Card B mounts at E:, the executor's only validation is `sourceDir.exists()`. If E: now points to Card B and Card B coincidentally has a file at the same relative path (e.g., another Canon card with `DCIM/100CANON/IMG_001.MOV`), robocopy copies from the WRONG source. Job marks completed; size verify (or even SHA-256, against B's bytes) passes; celebration card unlocks Erase, which then erases Card B. Card A's footage was never copied; Card B is now wiped.
- **Trigger**: Pause job on Card A ‚Üí eject A ‚Üí insert B into same slot ‚Üí resume. The Kingston SD hub the team uses makes letter-remap routine when cards swap.
- **Why prior rounds missed**: round-25 added `Platform.localHostname` cross-machine NAS guards; `eraseDrive` re-verifies WMI serial. The TRANSFER-time path has NO drive-identity check at all.

### F-2 ‚Äî Erase wipes content added to SD card after job enumeration

- **Opus**: P1-2 CERTAIN ¬∑ **Codex**: P1 #2 CERTAIN
- **Files**:
  - `lib/ui/widgets/erase_drive_action.dart:103-108` (eligibility checks planned files only)
  - `lib/services/drive_service.dart:260` (`Get-ChildItem | Remove-Item` deletes ALL children)
  - `lib/services/job_queue_service.dart:1397-1431` (enumeration is one-shot snapshot at job creation)
- **Failure mode**: Enumeration is a one-shot snapshot at batch creation. If the camera operator (or app operator) writes files to the SD card AFTER the job is queued but BEFORE it runs, those files are NOT in the planned set and will NEVER be transferred. `eraseEligibilityReason()` only checks `job.status == completed` AND every PLANNED file is `verified=true`. It does NOT compare current SD card contents against the planned set. Erase button enables; typed gate passes (correct path, correct serial); the card gets wiped. New files are gone.
- **Trigger**: Operator queues a batch ‚Üí camera flushes a final clip 30 seconds later ‚Üí batch runs ‚Üí operator clicks Erase. Most likely real-world vector.
- **Why prior rounds missed**: enumeration treated as authoritative; no "post-transfer reconciliation" pass exists. Failure is in the WORKFLOW assumption, not in any single function.

### F-3 ‚Äî Source-side symlinks/junctions cause copy loops or destination overwrites

- **Opus**: P1-3 LIKELY ¬∑ **Codex**: P2 SPECULATIVE (third bullet)
- **Files**:
  - `lib/services/drive_service.dart:127, :291` ‚Äî `dir.list(recursive: true)` defaults to `followLinks: true`
  - `lib/services/job_queue_service.dart:1400` ‚Äî same default in batch enumeration
- **Failure mode**: 017B added DEST-side `FileSystemEntity.type(..., followLinks: false)` checks. The SOURCE-side mirror was never added. SD cards from non-Windows formatters (rare in honest workflows but possible) can host symlinks/junctions. If the source has a junction `E:\backup ‚Üí D:\my-old-shoots`, `Directory.list(recursive: true)` walks INTO the junction target and treats those files as if they were on the SD card. A junction at source pointing INTO the destination tree creates an enumeration cycle (no cycle detection in `dart:io`).
- **Trigger**: Symlinked SD card (rare but real attack surface).
- **Why prior rounds missed**: 017B documented dest-side guard; source side silent.

## CONVERGENT P2 (correctness regression)

### F-4 ‚Äî `forceDestDeleteApproved` consumed BEFORE delete + robocopy actually succeed

- **Opus**: P2-2 LIKELY ¬∑ **Codex**: P2 #1 CERTAIN
- **Files**: `lib/services/job_queue_service.dart:705-709, :765`
- **Failure mode**: At the top of the per-file iteration, `forceDestDelete = file.forceDestDeleteApproved` is read AND cleared via `clearForceDestDeleteApproved`. Then dest is deleted, then robocopy runs. If the operator cancels mid-robocopy or the app crashes between consumption and robocopy success, the persisted state is `pending, forceDestDeleteApproved=false` ‚Äî the operator's retry intent is GONE. The next pass re-hits the same corrupt destination without forcing replacement. CLAUDE.md spec says "Re-mismatch on next pass requires fresh banner Retry click" ‚Äî code does not match this guarantee under cancel-mid-operation.
- **Why prior rounds missed**: round-2 P2 #2 fix added persistence across app restart between Retry click and consumption. It didn't audit consumption ordering for the cancel-during-consumption window.

### F-5 ‚Äî HandBrake compression has no staging dir / no partial-output cleanup

- **Opus**: P2-4 LIKELY ¬∑ **Codex**: P2 #2 LIKELY
- **Files**:
  - `lib/services/compression_service.dart:34-36`
  - `lib/services/job_queue_service.dart:1196-1278` (no preflight conflict check, no staging)
- **Failure mode**: Unlike transfer (robocopy with `/XN /XC /XO` + executor-side delete + staging dir), compression simply calls `HandBrakeCLI -i input -o output --preset name` directly. No pre-flight check, no staging dir, no orphan sweep. If a prior HandBrake run was killed mid-encode (operator cancel, crash, OOM), a partial `.mp4` sits at the destination. On retry HandBrake CLI may overwrite OR refuse depending on version. Worse: the partial encoded file sits at the destination forever, looking like real output to anyone browsing the folder. Operator opens it expecting real footage, gets a corrupted/truncated file. The startup sweep only handles `.tmp_robocopy_*`, not partial HandBrake outputs. No equivalent of `wasOverwriteApproved` for compression ‚Äî operator's intent for output overwrite is implicit.
- **Why prior rounds missed**: 015 / 016 / 017A / 017B / 018 all focused on the transfer pipeline because that's where source-erase hangs off. Compression treated as a post-stage that's "safe" because output goes to operator-chosen folder.

## DIVERGENT findings (single auditor, lower confidence ‚Äî may bundle)

### Codex-only

- **F-D1** [P1 SPECULATIVE]: Size-mode + renamed/staged transfer where the FINAL target already exists with same length but different bytes ‚Üí could pass verify. (`lib/services/transfer_service.dart:163-165`, `lib/services/job_queue_service.dart:1035`)

- **F-D3** [P2 SPECULATIVE]: Startup sweep matches the bare `.tmp_robocopy_*` prefix and deletes any marker-less directory whose name starts with that. Operator-created dir with that prefix would be destroyed.

### Opus-only

- **F-D4** [P2 LIKELY]: Cross-machine NAS write race for LIVE transfers. Round-25 only addressed the SWEEP cross-machine case, not concurrent live writes to same NAS dest from two machines.
- **F-D5** [P2 SPECULATIVE]: `Job.createdAt` mtime cutoff inverts under DST fall-back / NTP backward jump. Foreign intrusion modified at `mtime < createdAt` (post clock-jump) ‚Üí mtime guard says "not a TOCTOU intrusion" ‚Üí robocopy `/XN /XC /XO` skips ‚Üí size-mode passes the foreign content.
- **F-D6** [P2 LIKELY]: Long-path (>260 char) SHA-256 hashing on PS 5.1 fails permanently ‚Üí operator stuck in `verifyStatus=unverified` retry loop forever. Round-22 mentioned `\\?\` probing; only the warning shipped, not the runtime prefix.
- **F-D7** [P3]: `drive_service._runPowerShell` doesn't enforce length-3 argv invariant (open footgun for future refactors that re-add a 4th argv element ‚Äî re-opens the v2.4.0 root cause for these specific helpers).
- **F-D8** [P3 SPECULATIVE]: `eraseDrive`'s `Remove-Item` doesn't use `-LiteralPath` ‚Äî wildcard chars in filenames could be misinterpreted (rare on camera-formatted cards).

## Why this audit was needed

The 25 prior Codex rounds were scoped to feature deltas. Each round asked "does this CHANGE introduce a regression?" ‚Äî none asked "does the CHAIN from create ‚Üí enumerate ‚Üí transfer ‚Üí verify ‚Üí erase have workflow-level invariants no single feature owns?" Both holistic auditors landed on the same workflow-level holes: the create‚Üíenumerate‚Üíerase chain assumes the SD card content is stable across the chain (F-2), the transfer path assumes the drive letter binds to the same physical card (F-1), the source-side hardening was never mirrored from dest-side (F-3), the forceDestDelete state machine was never audited for cancel-mid-consumption (F-4), and the compression pipeline was never given the staging-dir safety net the transfer pipeline got (F-5).

This is exactly the value of a fresh holistic-framing audit after a long incremental-review history.
